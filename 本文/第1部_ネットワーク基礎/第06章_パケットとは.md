# 第06章 パケットとは

**― 通信データを運ぶ小さな単位 ―**

> この章では、データを分割して制御情報を付ける理由と、パケットを観察して障害を調べる基本を学びます。

------------------------------------------------------------------------

# 1. この章で学べること

- パケットに分割する理由
- ヘッダとペイロードの役割
- フレーム、パケット、セグメントの違い

# 2. この章の位置付け

第2章で通信の流れ、第3・4章で階層、第5章で転送する機器を学びました。本章では、実際に運ばれるデータ単位へ注目します。

# 3. なぜパケットが必要になったのか

大きなファイルを一つの塊で送ると、その間は回線を占有し、途中で誤りが起きれば全体を送り直す必要があります。ネットワークごとに一度に運べるサイズにも上限があります。

そこでデータを小さく分割し、複数の通信で回線を共有しながら運びます。失われた部分だけを再送でき、途中の機器も一定の単位で処理できます。

# 4. パケットの概要

**パケット（Packet）**は、ネットワークで転送されるデータの単位です。通常は制御情報である**ヘッダ（Header）**と、運びたい内容である**ペイロード（Payload）**から構成されます。

| 部分 | 例 | 役割 |
|---|---|---|
| Ethernetヘッダ | 送信元・宛先MACアドレス | 同一リンク内で届ける |
| IPヘッダ | 送信元・宛先IPアドレス | ネットワーク間で届ける |
| TCPヘッダ | ポート番号、シーケンス番号 | 順序や再送を管理する |
| ペイロード | HTTPなどのデータ | アプリケーションの情報を運ぶ |

# 5. 詳しい仕組み

## 呼び名の違い

- トランスポート層のTCPデータ単位：**セグメント（Segment）**
- インターネット層のIPデータ単位：パケット
- データリンク層のデータ単位：**フレーム（Frame）**

会話ではまとめて「パケット」と呼ぶこともあります。正確な調査では、どの層の単位かを区別します。

## MTU

**最大転送単位（Maximum Transmission Unit: MTU）**は、一つのリンクで運べるIPパケットサイズの上限です。Ethernetでは1500バイトがよく使われますが、必ず1500とは限りません。

経路上のMTUに対してパケットが大きすぎると、分割や送信失敗が起こります。IPv4の分割を**フラグメンテーション（Fragmentation）**と呼びます。一般には送信側が適切なサイズに調整し、不要な分割を避けます。

## TTL

**生存時間（Time To Live: TTL）**は、IPパケットがルータを通過するたびに減る値です。0になると破棄され、誤った経路でパケットが永久に循環することを防ぎます。

## チェックサム

**チェックサム（Checksum）**は、転送中のデータ破損を検出するための値です。層やプロトコルにより対象範囲と扱いは異なります。誤りを検出することと、自動的に訂正することは別です。

# 6. Linuxではどうなるか

**tcpdump**は、インターフェースを通るパケットを取得・表示するコマンドです。

```bash
# 利用可能なインターフェースを表示
sudo tcpdump -D

# DNS通信を名前解決せずに5件表示
sudo tcpdump -i any -nn -c 5 port 53

# ICMPをファイルへ保存
sudo tcpdump -i any -nn icmp -w ping.pcap
```

取得ファイルにはIPアドレス、ホスト名、平文のアプリケーションデータなどが含まれる可能性があります。業務環境では権限、保存場所、共有範囲、削除方法を事前に確認します。

MTUは次のように確認できます。

```bash
ip link show
ip route get 192.0.2.10
```

# 7. 実務ではどう使われるか

## 実務コラム：小さな通信は成功するが大きな通信が止まる

VPNやトンネルを追加するとヘッダ分だけ利用可能サイズが小さくなります。経路上で必要な通知が遮断されると、接続は成立するのに大きなデータだけ止まることがあります。

```bash
ping -c 3 192.0.2.10
ping -M do -s 1400 -c 3 192.0.2.10
ip link show
```

`-M do` や `-s` の対応はping実装により異なります。サイズを段階的に変え、MTUやトンネル設定を確認します。

# 8. FE/APではどう問われるか

ヘッダとペイロード、カプセル化、MTU、フラグメンテーション、TTL、各層のデータ単位が問われます。値の暗記だけでなく、回線共有、ループ防止、誤り検出という目的と結び付けます。

# 9. まとめ

- データを小さく分けることで回線を共有し、必要部分だけ再送できます。
- ヘッダは配送や制御の情報、ペイロードは運ぶ内容です。
- パケット取得は強力ですが、機密情報を含む可能性があるため慎重に扱います。

# 10. 理解度チェック

1. ヘッダとペイロードの違いを説明してください。
2. TTLは何のためにありますか。
3. MTUの問題ではどのような現象が起こり得ますか。

# 11. 解答・解説

## 問1

ヘッダは宛先や順序など通信制御に必要な情報、ペイロードはアプリケーションなどが実際に運びたいデータです。

## 問2

ルータを通るたびに値を減らし、経路ループが発生してもパケットが永久に残らないようにします。

## 問3

小さな通信は成功するのに大きなデータだけ停止する、VPN経由だけ失敗する、といった現象があります。経路に適したサイズへ調整できていない可能性を調べます。

# 12. 実務で考えてみよう

## ケース：tcpdumpにはパケットが来ているがアプリケーションは応答しない

### 解答例

宛先ポート、TCPフラグ、再送、応答パケットの有無を確認します。同時に `ss -lntp` でサービスの待受状態、アプリケーションログ、ホストFirewallを確認します。「パケットが一つ見えた」だけでは双方向通信の成立を意味しません。
