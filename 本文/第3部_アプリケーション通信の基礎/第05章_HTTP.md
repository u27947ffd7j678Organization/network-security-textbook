# 第05章 HTTP

**― Webの要求と応答を読む ―**

> この章では、HTTPが必要な理由を中心に学びます。

------------------------------------------------------------------------

# 1. この章で学べること

- HTTPが必要な理由
- リクエストとレスポンスの構造
- 主要メソッドとステータスコード
- ヘッダとボディの役割
- curlでHTTP通信を確認する方法

# 2. この章の位置付け

TCPは信頼性のある通信路を提供しますが、Webページを要求する表現までは決めません。本章では、その通信路上でWebの要求と応答を定めるHTTPを扱います。

# 3. なぜこの技術が必要になったのか

ブラウザとサーバが異なる実装でも、欲しい資源、処理方法、結果を共通形式で伝える必要があります。その規則がHTTPです。

# 4. 技術の概要

**HTTP（Hypertext Transfer Protocol）**は、クライアントがリクエストを送り、サーバがレスポンスを返すアプリケーションプロトコルです。HTTP自体は状態を保持しない設計で、必要な状態はCookieやセッションなど別の仕組みで補います。

# 5. 詳しい仕組み

## リクエストとレスポンス

```text
GET /index.html HTTP/1.1
Host: www.example.com
Accept: text/html

HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 1256

<html>...</html>
```

開始行、ヘッダ、空行、必要に応じたボディで構成されます。

## 主なメソッド

| メソッド | 主な意味 |
|---|---|
| GET | 資源を取得 |
| HEAD | ボディなしでヘッダを取得 |
| POST | データを送って処理を依頼 |
| PUT | 指定資源を作成・置換 |
| DELETE | 指定資源を削除 |

実際の許可や安全性はサーバ実装によります。GETだから機密情報をURLへ含めてよいわけではありません。

## ステータスコード

- 1xx：処理中
- 2xx：成功
- 3xx：転送
- 4xx：クライアント側の要求に関する問題
- 5xx：サーバ側の処理に関する問題

`404 Not Found` はHTTP応答を受信できているため、物理接続やTCPよりもURLやアプリケーションを優先して調べます。

## HTTPバージョン

HTTP/1.1、HTTP/2、HTTP/3では転送方法が異なります。HTTP/3はQUICを使いUDP上で動作しますが、メソッドやステータスコードというHTTPの意味は維持されます。

# 6. Linuxではどうなるか

```bash
# レスポンスヘッダを表示
curl -I http://www.example.com/

# リクエストと応答の詳細を表示
curl -v http://www.example.com/

# ステータスと時間だけ表示
curl -sS -o /dev/null -w '%{http_code} %{time_total}\n' http://www.example.com/
```

代表的な出力例（必要な部分のみ抜粋）

```text
$ curl -v http://www.example.com/
> GET / HTTP/1.1
> Host: www.example.com
< HTTP/1.1 301 Moved Permanently
< Location: https://www.example.com/

$ curl -sS -o /dev/null -w '%{http_code} %{time_total}\n' http://www.example.com/
301 0.042
```

確認ポイント

- `>` は送信したリクエスト、`<` は受信したレスポンスです。
- `301` は恒久的転送、`Location` が転送先です。
- `time_total` は処理全体の秒数で、どの段階が遅いかは追加項目で分けて測定します。

# 7. 実務ではどう使われるか

## 実務コラム：502 Bad Gateway

リバースプロキシが背後のアプリケーションから正常な応答を得られない場合などに502が返ります。

```bash
curl -vI https://service.example.com/
```

代表的な出力例（必要な部分のみ抜粋）

```text
< HTTP/2 502
< server: reverse-proxy
```

確認ポイント

- HTTP応答を返した機器と背後の接続を分けて確認します。
- 5xxでも原因が必ず最終アプリケーションだけにあるとは限りません。

# 8. FE/APではどう問われるか

HTTPの要求・応答、メソッド、ステータスコード分類、ヘッダ、Cookie、HTTPバージョンが問われます。コードを障害切り分けへ結び付けます。

# 9. まとめ

- HTTPはWebの要求と応答を定めます。
- メソッドが操作、ステータスコードが結果を表します。
- HTTPエラーを受け取れれば通信がHTTP層まで進んでいます。

# 10. 理解度チェック

1. HTTPリクエストを構成する主な要素を答えてください。
2. 3xxと5xxは何を表しますか。
3. HTTPが状態を持たないとはどういう意味ですか。

# 11. 解答・解説

## 問1

リクエスト行、ヘッダ、空行、必要に応じたボディです。

## 問2

3xxは転送、5xxはサーバ側処理に関する問題です。

## 問3

各要求を独立して扱い、以前の要求状態をHTTP自体が自動的に保持しないことです。

# 12. 実務で考えてみよう

## ケース：ブラウザだけ古いページが表示される

### 解答例

ブラウザ、プロキシ、CDNのキャッシュを区別し、Cache-Control、Age、ETagなどのヘッダを確認します。サーバへ直接届いた応答と比較します。
