# 第02章 ポート番号

**― 一台の端末にあるサービスの窓口 ―**

> この章では、ポート番号が必要な理由とIPアドレスとポート番号の役割分担を中心に学びます。

------------------------------------------------------------------------

# 1. この章で学べること

- ポート番号が必要な理由
- IPアドレスとポート番号の役割分担
- ウェルノウンポートと動的ポート
- 待受ソケットと接続の識別
- Linuxでポート使用状況を調べる方法

# 2. この章の位置付け

IPアドレスは端末やインターフェースへの届け先を示します。本章では、その端末上で通信を受け取るアプリケーションを区別するポート番号を扱います。

# 3. なぜこの技術が必要になったのか

一台のサーバではWeb、SSH、メールなど複数のサービスが同時に動きます。IPアドレスだけでは、届いたデータをどのプロセスへ渡すか判断できません。そのためTCPとUDPはポート番号を使います。

# 4. 技術の概要

**ポート番号（Port Number）**は0〜65535の16ビット値です。通信はプロトコル、送信元IP・ポート、宛先IP・ポートの組合せで識別されます。TCPの53番とUDPの53番は別の窓口です。

# 5. 詳しい仕組み

## ポート番号の範囲

| 範囲 | 名称 | 主な用途 |
|---:|---|---|
| 0〜1023 | ウェルノウンポート | HTTP 80、HTTPS 443、SSH 22など |
| 1024〜49151 | 登録済みポート | 登録されたアプリケーションなど |
| 49152〜65535 | 動的・プライベートポート | クライアントの一時ポートなど |

範囲は目安であり、ポート番号だけでアプリケーションを断定できません。設定で変更でき、別のプロトコルが使う場合もあります。

## サーバ側とクライアント側

サーバは決めたポートで**待ち受け（Listen）**ます。クライアントは通常、一時的な**エフェメラルポート（Ephemeral Port）**を送信元に選びます。

```text
クライアント 192.0.2.10:52144
        ↓
サーバ       198.51.100.20:443
```

同じサーバの443番へ多数のクライアントが接続しても、送信元IP・ポートが異なるため区別できます。

## バインドするアドレス

`127.0.0.1:8080` は端末内部からだけ、`0.0.0.0:8080` はすべてのIPv4インターフェースで待ち受ける指定です。公開範囲はFirewallと合わせて判断します。

# 6. Linuxではどうなるか

```bash
# 待受中のTCP/UDPとプロセスを表示
ss -lntup

# 443番を使用するソケットだけ表示
ss -tnp '( sport = :443 or dport = :443 )'

# ポート名の対応を確認
getent services https
```

代表的な出力例（必要な部分のみ抜粋）

```text
$ ss -lntup
Netid State  Local Address:Port Process
tcp   LISTEN 0.0.0.0:22        users:(("sshd",pid=742,fd=3))
tcp   LISTEN 127.0.0.1:8080    users:(("app",pid=900,fd=7))
udp   UNCONN 0.0.0.0:5353      users:(("mdns",pid=600,fd=4))

$ getent services https
https 443/tcp
```

確認ポイント

- `LISTEN` はTCP待受、`UNCONN` はUDPでよく表示される状態です。
- `0.0.0.0:22` と `127.0.0.1:8080` では外部から到達できる範囲が異なります。
- `users` でポートを使用するプロセスを確認できますが、権限により省略されます。
- `443/tcp` はサービス名の標準的な対応で、実際の通信内容の保証ではありません。

# 7. 実務ではどう使われるか

## 実務コラム：Address already in use

サービス起動時の「Address already in use」は、同じアドレス・ポートを別プロセスが使用している場合に発生します。

```bash
ss -lntp 'sport = :8080'
```

代表的な出力例（必要な部分のみ抜粋）

```text
$ ss -lntp 'sport = :8080'
LISTEN 0 128 0.0.0.0:8080 0.0.0.0:* users:(("old-app",pid=900,fd=7))
```

確認ポイント

- プロセス名とPIDを確認し、正規のサービスか調べます。
- 直ちに停止せず、依存関係と利用者影響を確認します。

# 8. FE/APではどう問われるか

代表的なウェルノウンポート、TCPとUDPで別空間であること、クライアントの一時ポート、ソケットの識別が問われます。番号は役割と合わせて理解します。

# 9. まとめ

- ポート番号は端末内のアプリケーションの窓口を区別します。
- サーバは既知のポートで待ち受け、クライアントは一時ポートを使うのが一般的です。
- 待受アドレスにより外部から到達できる範囲が変わります。

# 10. 理解度チェック

1. ポート番号は何ビットで、範囲はいくつですか。
2. 同じサーバの443番へ複数クライアントが接続できる理由を説明してください。
3. `127.0.0.1:8080` と `0.0.0.0:8080` の違いは何ですか。

# 11. 解答・解説

## 問1

16ビットで、0〜65535です。

## 問2

送信元IP・送信元ポート・宛先IP・宛先ポート・プロトコルの組合せが異なるためです。

## 問3

前者はループバックだけ、後者はすべてのIPv4インターフェースで待ち受けます。

# 12. 実務で考えてみよう

## ケース：サービスはLISTENだが別端末から接続できない

### 解答例

待受アドレスがループバックだけでないか、サーバのIPアドレス、経路、ホストとネットワークのFirewallを確認します。LISTENはプロセスの待受を示すだけで、経路全体の到達性を保証しません。
