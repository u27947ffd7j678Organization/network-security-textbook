# 第07章 SSH

**― Linuxサーバを安全に遠隔操作する ―**

> この章では、SSHが必要な理由を中心に学びます。

------------------------------------------------------------------------

# 1. この章で学べること

- SSHが必要な理由
- パスワード認証と公開鍵認証
- ホスト鍵によるサーバ確認
- SSH接続とポートフォワーディング
- scp・sftpとの関係

# 2. この章の位置付け

HTTPSはWeb通信をTLSで保護します。本章では、Linuxサーバの遠隔ログインや管理通信を暗号化するSSHを扱います。

# 3. なぜこの技術が必要になったのか

平文の遠隔ログインでは、認証情報や操作内容が盗聴されます。また、接続先が本物のサーバか確認する必要があります。SSHは暗号化された通信路と相互の認証手段を提供します。

# 4. 技術の概要

**SSH（Secure Shell）**は、遠隔ログイン、コマンド実行、トンネル、ファイル転送に使われるプロトコルです。通常はTCP 22番を使います。クライアントはサーバのホスト鍵を確認し、利用者はパスワードや公開鍵で認証します。

# 5. 詳しい仕組み

## ホスト鍵

初回接続時、クライアントはサーバのホスト鍵フィンガープリントを確認します。以後は `known_hosts` と比較し、鍵が突然変われば警告します。サーバ再構築など正当な変更か、中間者攻撃かを確認せず警告を消してはいけません。

## 公開鍵認証

利用者は秘密鍵を安全に保持し、対応する公開鍵をサーバの `authorized_keys` へ登録します。秘密鍵自体をサーバへ送るわけではありません。秘密鍵にはパスフレーズを設定し、必要に応じてssh-agentを使います。

## パスワード認証

導入しやすい一方、弱いパスワードや総当たり攻撃のリスクがあります。要件に応じて公開鍵認証、多要素認証、接続元制限を組み合わせます。

## トンネル

SSHポートフォワーディングは、SSHの暗号化通信路を通して別のTCP通信を転送します。便利ですが、組織のアクセス制御を迂回しないよう許可と用途を確認します。

# 6. Linuxではどうなるか

```bash
# 詳細ログ付きで接続
ssh -v user@server.example.com

# 公開鍵を作成
ssh-keygen -t ed25519 -C 'admin@example.com'

# 接続と待受を確認
ss -tnp 'dport = :22'
```

代表的な出力例（必要な部分のみ抜粋）

```text
$ ssh -v user@server.example.com
debug1: Connecting to server.example.com [192.0.2.50] port 22.
debug1: Server host key: ssh-ed25519 SHA256:AbCd...
debug1: Authentications that can continue: publickey,password
debug1: Authentication succeeded (publickey).

$ ss -tnp 'dport = :22'
ESTAB 192.0.2.10:53022 192.0.2.50:22 users:(("ssh",pid=1800,fd=3))
```

確認ポイント

- `Connecting` で名前解決結果とポートを確認します。
- ホスト鍵フィンガープリントを管理情報と照合します。
- `Authentication succeeded (publickey)` は公開鍵認証の成功です。
- `ESTAB` はTCP接続が確立している状態です。

# 7. 実務ではどう使われるか

## 実務コラム：ホスト鍵が変わった

```text
WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!
```

この警告では接続を止め、サーバ管理者へ正しいフィンガープリントを別経路で確認します。再構築など正当な理由を確認後、対象行だけを更新します。

```bash
ssh-keygen -F server.example.com
```

代表的な出力例（必要な部分のみ抜粋）

```text
# Host server.example.com found: line 12
server.example.com ssh-ed25519 AAAAC3...
```

確認ポイント

- 登録済み鍵の種類と行を確認し、無関係なknown_hosts全体を削除しません。

# 8. FE/APではどう問われるか

SSHの目的、ホスト鍵と利用者鍵の違い、公開鍵認証、TCP 22番、scp・sftpが問われます。秘密鍵を共有しない原則が重要です。

# 9. まとめ

- SSHは遠隔操作と関連通信を暗号化します。
- ホスト鍵はサーバ、利用者の公開鍵認証はユーザーを確認します。
- 警告を無視せず、フィンガープリントを別経路で確認します。

# 10. 理解度チェック

1. ホスト鍵と利用者の公開鍵は何を認証しますか。
2. 公開鍵認証で秘密鍵をサーバへ送らないのはなぜですか。
3. SSHホスト鍵変更警告が出たときの対応を説明してください。

# 11. 解答・解説

## 問1

ホスト鍵は接続先サーバ、利用者の公開鍵はログインするユーザーを認証します。

## 問2

秘密鍵は本人だけが保持する認証の根拠であり、漏えいすると成りすましに使われるためです。

## 問3

接続を止め、管理者から正しいフィンガープリントを別経路で確認し、正当な変更時だけ対象鍵を更新します。

# 12. 実務で考えてみよう

## ケース：SSH接続だけ失敗する

### 解答例

名前解決と経路、TCP 22番の到達性、sshdの待受、Firewall、認証ログを順に確認します。ping成功だけではSSHの正常性を保証しません。
